# 莫奈的画境 / Monet's Dreamscape

印象派风格横版卷轴探索游戏 — 纯 HTML5 Canvas 实现。探索莫奈的六幅名画，收集隐藏的宝物，见证莫奈与卡米耶的爱情故事。

## 简介

玩家操控一个发光的精灵，在以莫奈真实画作为蓝本的世界中冒险。每个关卡都有独特的游戏机制，灵感来自画作主题 — 从灯光摇曳的巴黎画室，到风吹花舞的花园，再到撑着阳伞滑翔过山丘。

游戏按时间线讲述莫奈与卡米耶·东西欧的爱情故事，配有英文语音旁白和中英双语字幕，从他们在巴黎画室的初遇，延续到莫奈晚年在吉维尼的岁月。

## 功能特性

- **6个关卡** — 基于莫奈真实画作，每关独特游戏机制
- **中英双语** — 菜单和游戏内均可实时切换
- **语音旁白** — 莫奈与卡米耶的对话（42条语音，英文TTS + 中英字幕）
- **画作渐现** — 背景从灰色素描逐渐变为全彩，随收集进度展现
- **程序化音乐** — Web Audio API 实时生成环境音乐，无需音频文件
- **收集系统** — 每关3个颜料管钥匙 + 5个纪念品，持久化图鉴
- **画作画廊** — 查看已解锁的莫奈名画及介绍
- **AI生成精灵** — 8帧角色动画，运行时自动去除黑色背景
- **3层视差滚动** — 雾气、光柱、暗角、噪点等氛围特效
- **移动端适配** — 响应式布局 + 虚拟触控按钮

## 架构亮点

### 单文件游戏引擎
所有游戏逻辑集中在 `game.js` 一个文件中 — 物理引擎、渲染、关卡数据、国际化、音频、UI 状态管理。这是为了实现零依赖的静态托管部署。

### 精灵处理管线
AI 生成的角色贴图带有黑色背景，通过运行时两遍像素处理自动去除：
1. **第一遍**：扫描所有帧，找到内容边界框，计算最大统一尺寸
2. **第二遍**：将每帧居中放入统一画布，移除暗色像素并做渐变透明过渡

确保所有动画帧尺寸一致，防止状态切换时的抖动。

### 画作渐现系统
三种揭示模式增加关卡间的视觉多样性：
- **全局渐变** — 整个背景从去饱和逐渐变为全彩
- **局部扩散** — 颜色从每个收集点向外辐射
- **分层揭示** — 水平区域从下到上逐层显现

### 程序化音乐
不包含任何音乐文件，通过 Web Audio API 实时生成环境音乐：
- 第1层：和弦铺底（持续的和声基础）
- 第2层：五声音阶旋律（随机生成）
- 第3层：低音（和弦根音）
- 第4层：滤波白噪声纹理

## 设计理念

视觉设计取材于美术馆美学 — 暖色羊皮纸色调（#f5ead0）用于文字，深色画廊黑（#0a0a12）用于背景。Georgia 衬线字体呼应古典艺术图录的气质。大量运用透明度叠加和模糊滤镜，再现印象派绘画特有的光影朦胧感。

叙事沿莫奈与卡米耶的真实时间线展开，将平台跳跃机制植根于情感背景之中，而非纯粹的难度堆砌。

## 开发历程

本项目完全通过 AI 辅助编程开发 — 从概念到部署，经历了 35 个迭代版本。以下是详细的开发过程、遇到的问题及解决方案。

### 第一阶段：原型开发（V1–V5）

**目标**：实现一个可运行的单关卡平台跳跃游戏。

项目始于一个简单的设想：如果你能*走进*一幅莫奈的画作会怎样？第一个版本是一个极简的 HTML5 Canvas 游戏：
- 单关卡（睡莲）配静态平台
- 程序化绘制的角色（彩色矩形）
- 基础的左右移动和重力系统
- 一张背景图片

**关键决策**：从一开始就将所有代码放在单个 `game.js` 文件中。这不是偷懒 — 游戏需要部署在会剥离内联脚本和样式的静态托管平台（mule.page）上。单个外部 JS 文件是最可靠的方案。

**V3–V5** 加入了核心游戏循环：可收集的颜料管作为"钥匙"解锁关卡终点、简单的 HUD、碰撞检测。物理系统经历了多次迭代 — 早期版本中玩家会从平台上滑落或在高速移动时穿过平台。修复方法是在速度过高时将碰撞检测拆分为更小的子步骤。

### 第二阶段：三关卡与音频（V6–V12）

**目标**：扩展到三个关卡，各具独特氛围，添加音乐和语音旁白。

初始版本选择了三幅莫奈画作：
1. 《睡莲》(1906) — 水平布局，宁静，漂浮的平台
2. 《日出·印象》(1872) — 港口场景，船只作为平台
3. 《日本桥》(1899) — 垂直花园，桥为核心元素

**背景美术**使用 MuleRouter API（Google Nano Banana 模型）生成 — 以每幅画作为风格参考，生成该画作调色板和笔触风格的宽幅全景背景。

**程序化音乐**是刻意选择而非使用预录音频。Web Audio API 通过四个层次实时生成环境音乐（和弦铺底、五声音阶旋律、低音、滤波噪声）。这完全消除了对音乐文件的需求，保持部署轻量化。每个关卡有自己的调性和节奏以匹配氛围。

**语音旁白**使用 edge-tts（Microsoft Edge TTS，免费，无需 API Key）生成。对话通过位置触发讲述莫奈与卡米耶的故事 — 玩家在关卡中行走时会听到两人的对话。英文配音配合中英双语字幕。

**i18n 国际化系统**从零构建 — 扁平的键值字典配合 `t('key')` 查找，支持运行时切换语言。每一段 UI 文字、语音字幕和物品名称都通过此系统管理。

### 第三阶段：打磨与系统完善（V13–V20）

**目标**：添加收集机制、UI 打磨和纪念品图鉴。

这个阶段专注于让游戏有完整感：

- **纪念品系统**：每关 5 个可选收集品（调色刀、画笔、信件、花朵等）。每个都是 AI 生成的图片，带有名称和描述。收集状态持久化存储在 localStorage 中，出现在图鉴里。
- **背包 UI**：按 Tab/B 键查看当前关卡已收集的物品，以抽屉式面板展示。
- **画作展示**：通关后展示莫奈原作，附带标题、画家、年份和描述。既是奖励，也是艺术教育。
- **暂停系统**：ESC 暂停游戏，显示毛玻璃效果遮罩。可继续、返回菜单或重新开始。
- **3层视差滚动**：背景、中景和前景以不同速度滚动，营造纵深感。搭配动态雾气、光柱、暗角和噪点颗粒增强氛围。
- **移动端控制**：虚拟方向键和跳跃按钮，触屏设备自动检测显示。

### 第四阶段：扩展至6关（V21–V28）

**目标**：通过6个按时间排列的关卡讲述完整的莫奈-卡米耶故事。

原有的3个关卡覆盖了莫奈晚年的吉维尼时期。为了讲述完整的爱情故事，新增了3个早期关卡并重新排序：

| 新顺序 | 画作 | 新增内容 |
|--------|------|----------|
| L1（新） | 穿绿裙的女人, 1866 | **光影机制** — 玩家携带灯笼，光照范围外的平台不可见 |
| L2（新） | 花园中的女人们, 1866 | **风力机制** — 周期性阵风推动玩家，平台摇摆 |
| L3（新） | 撑阳伞的女人, 1875 | **阳伞滑翔** — 长按跳跃可滑翔，上升气流区域用于垂直移动 |
| L4（原L1） | 睡莲, 1906 | 添加了消失的莲叶平台 |
| L5（原L2） | 日出·印象, 1872 | **潮汐机制** — 水位按12秒周期升降 |
| L6（原L3） | 日本桥, 1899 | **季节变换** — 背景从春→夏→秋过渡 |

每个新关卡需要：
- 背景图生成（MuleRouter API）
- 7条语音旁白（edge-tts）+ 中英字幕
- 5张纪念品图片（AI 生成）
- 高清画作图用于通关展示
- 关卡布局数据（平台、收集品位置、触发区域）
- 独特机制实现

**新平台类型**：新增10种 — `easel`（画架）、`shelf`（架子）、`frame`（画框）用于L1画室场景，`hedge`（树篱）、`bench`（长凳）、`branch`（树枝）用于L2花园，`grass`（草地）、`cloud`（云朵）、`kite`（风筝）用于L3山丘，`tidal`（潮汐平台）用于L5。

**画作渐现系统**：背景从灰色素描逐渐变为全彩，随收集进度展现。三种视觉模式增加多样性：
- *全局渐变*：整个背景从去饱和逐渐变为全彩（L2, L5, L6）
- *局部扩散*：颜色从每个收集点向外辐射，如墨水入水（L1, L4）
- *分层揭示*：水平区域从下到上逐层显现（L3）

**语音队列修复**：位置触发的对话可能在上一条还在播放时被触发，导致重叠。添加了 `busy` 标志和待播队列 — 新触发的语音会等当前语音播完再播放。

### 第五阶段：角色精灵与画廊（V29–V34）

**目标**：用 AI 生成的精灵替换程序化角色，添加画作画廊。

原始角色是程序化绘制的（彩色矩形加发光效果）。虽然功能正常，但与印象派艺术风格不协调。

**AI 精灵生成**：使用 MuleRouter API（Google Nano Banana 2 Edit 模型）生成 8 个角色姿态帧：
- idle, idle_look — 站立姿态
- walk_a, walk_b — 两帧行走循环
- jump_crouch, jump_apex — 跳跃起跳和最高点
- fall, landing — 下落和着地

这些被合成为单张精灵表（`sprite_sheet_v3.png`，2048x571）。

**黑色背景问题**：AI 生成的精灵带有黑色背景。构建了运行时像素处理管线来处理：
1. 第一遍：扫描全部8帧，找到内容边界框，计算最大统一画布尺寸
2. 第二遍：将每帧居中放入统一画布，移除暗色像素并做渐变透明过渡（亮度 < 25 → 全透明，25-60 → 渐变过渡）

这个两遍处理方案至关重要 — 早期版本独立处理每一帧，导致不同帧的边界框尺寸不同。角色在跳跃时会"缩小"，因为跳跃帧的宽高比和行走帧不同。统一归一化解决了这个问题。

**交叉淡入动画**：状态转换（待机→行走、行走→跳跃等）使用 250ms 的交叉淡入效果，防止生硬的瞬间切换。

**画作画廊**：新增菜单区域，以卡片形式展示全部6幅画作。已解锁的画作可点击查看全画及介绍。未解锁的画作显示为暗色预览加锁定图标。

**画廊模式隔离 Bug**：点击画廊中的画作复用了通关展示的遮罩层。"继续"按钮的事件监听会触发 `closePaintingReveal()`，该函数会进入下一关 — 所以点击画廊中的画作会意外开始游戏关卡。通过添加 `_galleryMode` 标志修复：为 true 时，关闭处理器返回画廊而非推进关卡。

### 第六阶段：画作展示重设计（V34–V35）

**目标**：消除画作展示遮罩中画作周围的黑边。

画作展示经历了3次设计迭代：

1. **V31**：固定尺寸容器 + `object-fit: cover` — 裁切竖幅画作，只显示下半部分。不可用。
2. **V33**：自适应布局 + `object-fit: contain` — 保持了宽高比，但画作周围出现黑色背景条。用户明确表示"不喜欢黑边"。
3. **V35（最终版）**：以画作为中心的方案 — 图片驱动容器尺寸。`max-height: 62vh; max-width: 80vw; width: auto; height: auto`，背景透明。通过 CSS 伪元素实现金色画框效果。没有固定容器，没有黑边。

遮罩层使用 `backdrop-filter: blur(24px)` 加深色半透明背景，画作以 `scale(1.08)→scale(1)` 的 cubic-bezier 曲线动画入场。

### 部署流程

游戏通过 Python 脚本（`deploy_v2.py`）部署到 mule.page（静态托管）：
1. 初始化与托管 API 的 MCP 会话
2. 创建新版本（V1, V2, ..., V35）
3. 为每个文件获取预签名上传令牌
4. 上传全部 90 个文件（HTML、CSS、JS、图片、音频）
5. 发布版本
6. 通过请求线上页面验证部署

每次部署约需 2 分钟（90个文件）。脚本包含验证步骤 — 检查线上页面响应中是否存在 v2 特有的 HTML 元素。

### 经验总结

1. **单文件架构的可扩展性出乎意料地好**。`game.js` 约4800行虽然很大但仍可导航。零构建步骤、即时部署的权衡对于这个体量的游戏是值得的。

2. **AI 生成的素材需要运行时处理**。不能直接使用 AI 生成的美术资源 — 黑色背景、不一致的尺寸、参差的质量都需要程序化处理。两遍精灵处理管线是最复杂的素材处理环节。

3. **迭代部署至关重要**。通过35个版本的线上部署和测试，在真实浏览器环境中发现 Bug，而非仅靠本地测试。自动化部署+验证流程使这成为可能。

4. **渐进式展示保持玩家参与度**。灰色到全彩的画作渐现给予了超越"收集所有物品"的实质进度感。玩家在字面意义上将大师之作重新绘就。

5. **叙事背景改变游戏体验**。同样的平台跳跃机制在爱情故事的背景下感受截然不同。在黑暗画室中收集物品*因为莫奈正在第一次画卡米耶*，创造了纯粹游戏性无法给予的情感投入。
